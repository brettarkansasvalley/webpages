# Toast Data DSL Queries

This directory contains Domain Specific Language (DSL) queries to replicate the CSV reports that were previously generated by Python scripts in `/data/reports/`.

## Overview

The DSL queries are executed against the JAQ JSON Loader server running at `http://localhost:3000`. This server provides a SQL-like interface for querying JSON data stored in its database.

## Directory Structure

```
DSL/
├── queries/          # DSL query files (.dsl.json)
├── outputs/          # Generated CSV outputs
├── scripts/          # Helper scripts
│   ├── run_dsl_query.sh      # Run a single query
│   └── run_all_queries.sh    # Run all queries
└── README.md         # This file
```

## DSL Syntax

The DSL supports SQL-like operations:

### Basic SELECT
```json
{
  "from": {
    "source_file": "labor_v1_employees.json",
    "alias": "emp"
  },
  "select": [
    {"expr": "emp.guid", "alias": "guid"},
    {"expr": "emp.firstName", "alias": "firstName"}
  ]
}
```

### JOINs
```json
{
  "from": {
    "source_file": "labor_v1_timeEntries_20260130.json",
    "alias": "te"
  },
  "joins": [
    {
      "source": {
        "source_file": "labor_v1_employees.json",
        "alias": "emp"
      },
      "on": {
        "left": "te.employeeReference.guid",
        "right": "emp.guid"
      },
      "join_type": "left",
      "skip_nulls": true
    }
  ]
}
```

### WHERE Clauses
```json
{
  "where": [
    {"field": "te.inDate", "op": "is_not_null"},
    {"field": "te.businessDate", "op": "=", "value": "20260130"}
  ],
  "where_op": "and"
}
```

### GROUP BY and Aggregations
```json
{
  "group_by": [
    {"field": "te.businessDate"},
    {"field": "te.employeeReference.guid"}
  ],
  "select": [
    {"expr": "count(te.guid)", "alias": "shifts_count"},
    {"expr": "sum(te.regularHours)", "alias": "regular_hours_total"}
  ]
}
```

### ORDER BY
```json
{
  "order_by": [
    {"field": "te.businessDate", "direction": "desc"},
    {"field": "te.inDate", "direction": "asc"}
  ]
}
```

## Limitations and Notes

### 1. Orders Data
The `orders_v2_orders_20260130.json` in the datalake contains only order GUIDs, not full order objects. Full order data with checks, payments, selections, etc. would need to be:
- Loaded separately from `/home/ubuntu/new_toasty/toasty/data/raw/orders/`
- Or queried from Toast API directly

### 2. Complex Aggregations
Some reports like `net_sales_by_employee_daily.csv` require complex aggregations across nested structures (orders → checks → payments → selections). These are challenging to express in the current DSL and may require:
- Pre-processing of the JSON data
- Custom aggregation functions
- Multiple query steps with intermediate results

### 3. Date Handling
The DSL doesn't have built-in date/time functions for:
- Parsing ISO8601 dates
- Converting between timezones
- Calculating hour differences
- Filtering by date ranges

These transformations need to be done in post-processing.

### 4. Array Handling
For nested arrays (e.g., `checks[].payments[]`), the DSL supports:
- `explode` to flatten arrays into rows
- `[]` notation for array field access in joins

## Migration from Python to DSL

### Python Approach (Current)
```python
# 1. Fetch data from Toast API
orders = get_orders_by_business_date(token, date)

# 2. Complex nested processing
for order in orders:
    for check in order.get("checks", []):
        for payment in check.get("payments", []):
            # Aggregate by employee
            ...

# 3. Write to CSV
write_csv_report(rows)
```

### DSL Approach (New)
```bash
# 1. Load JSON files into datalake (one-time)
curl -X POST http://localhost:3000/load

# 2. Query using DSL
curl -X POST http://localhost:3000/query/dsl \
  -H "Content-Type: application/json" \
  -d '{"query": "{...}"}'

# 3. Export to CSV
curl -X POST http://localhost:3000/export \
  -d '{"query": "{...}", "format": "csv"}'
```
